“What do we want by when?”

The Requirement analysis is the most important part of the SDLC as it affects all other phases. It is performed by the senior members of the team with inputs from:

  • customers
  • salespeople
  • industry experts
  • programmers
  • market surveys

The information from the requirement analysis is used to plan the basic project approach and to conduct a feasibility study which defines the various technical approaches to minimise risks associated with the project. Planning also involves It determines whether a project is doable, justifiable and worth the investment. This phase also involves quality assurance requirements which is a way of preventing mistakes and defects of products and avoiding problems when delivering services to customers.

TELOS is the acronym used to summarise the possible risks associated with the project. It stands for:

Testing is usually a subset of different parts of the SDLC, but the testing phase refers exclusively to testing where system flaws are reported, tracked, fixed and retested, until the product reaches the quality standards defined in the SRS. In this stage, testers will use the test cases they've developed to find flaws in the code and report to the bug tracking system. A developer will then solve the problem and send it back once the issue has been resolved. Spiral Model

The Spiral Model is the most flexible out of all the SDLC models. It combines the the iterative model's emphasis on repitition and the waterfall model. The spiral model goes through the planning, design, build and testing phases over and over, with gradual improvements at each pass. The software is developed in a serires of incremental releases. Each phase begins with the design goal and ends with client reviewing the result.

Agile Model.

The Agile SDLC model is based on the japanese KANBAN. It separates the product into cycles and delivers a working product very quickly. This methodology produces a succession of releases. Testing of each release feeds back info that’s incorporated into the next version. According to Robert Half, the drawback of this model is that the heavy emphasis on customer interaction can lead the project in the wrong direction in some cases. Iterative Model.

This SDLC model emphasizes repetition. Developers create a version very quickly and for relatively little cost, then test and improve it through rapid and successive versions. One big disadvantage here is that it can eat up resources fast if left unchecked. video vid2 span[style="display:none"] doesnt need the full requirement specification before the project starts, the process is repetitive, allowing new versions of the product for every cycle it is not suitable for small projects e.g. Rational Unified Process(RUP) developed by IBM. It is used when all the requirements are unknow.

V-Shaped Model.

An extension of the waterfall model, this SDLC methodology tests at each stage of development. As with waterfall, this process can run into roadblocks. Advantages introduces testing early simple, clear and easy suited for critical systems e.g. when people's life are at risk like medicine. military and hospital Disadvantages doesnt account for change so inflexible not suitable for small projects

Big Bang Model.

This high-risk SDLC model throws most of its resources at development and works best for small projects. It lacks the thorough requirements definition stage of the other methods.
